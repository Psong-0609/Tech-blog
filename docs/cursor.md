# Cursor — Tech-blog 프로젝트 진입점

Chat/Composer에서 작업할 때 **먼저 `docs/agent.md`**를 참고합니다. 여기서는 Cursor 사용자를 위한 요약과 저장소 관습만 적습니다. **포맷·린트 필수 절차는 아래 「작업 단위 완료 후」를 따릅니다.**

## 프로젝트 한 줄 요약

Next.js(App Router) 기반 기술 블로그 — 글은 저장소 루트 **`content/*.md`**(**`src/` 밖**, 형제 디렉터리), 편집은 **TipTap**·저장은 Markdown, 글 CRUD·카테고리·검색·다크모드 및 성능·**현재 `src/app` 중심 폴더 구조**·**하위 레이어 순수 함수 지향**·접근성 목표는 **`docs/agent.md`**에 정의되어 있습니다. **보안·가용성(모니터링·알림 등)은 추후** 다룹니다.

## 빠른 체크리스트 (변경 후)

| 영역     | 확인                                                                                                      |
| -------- | --------------------------------------------------------------------------------------------------------- |
| **품질** | **`pnpm format` 후 `pnpm lint`**를 순서대로 실행했는지(작업 한 묶음마다 필수)                             |
| 기능     | 작성/수정/삭제(루트 `content/*.md`, `src/` 외부, TipTap), 카테고리, 검색(카테고리+키워드), 테마 전환 동작 |
| 성능     | 불필요한 클라이언트 코드, 큰 이미지, 레이아웃 시프트                                                      |
| 구조     | 하위 유틸·변환은 순수 함수 지향, IO는 상위 경계에 모음                                                    |

## 로컬 명령

```bash
pnpm dev           # 개발 서버
pnpm build         # 프로덕션 빌드
pnpm format        # Prettier 적용 (작업 후 필수)
pnpm format:check  # Prettier 검사만 (CI·사전 확인)
pnpm lint          # ESLint (작업 후 필수)
```

## 작업 단위 완료 후 (필수)

코드·설정 등 **저장소에 반영되는 변경을 한 묶음 마칠 때마다** 프로젝트 루트에서 아래를 **순서대로** 실행한다.

1. **`pnpm format`** — Prettier 적용
2. **`pnpm lint`** — ESLint 검사

실패 시 수정 후 다시 실행한다. 포맷만 하거나 린트만 하고 생략하지 않는다. 검증만 필요할 때는 **`pnpm format:check`**와 **`pnpm lint`**를 함께 쓸 수 있다.

## 파일 역할

에이전트·Cursor 관련 문서는 **저장소 루트의 `docs/`** 에만 둔다.

- **`docs/agent.md`**: 에이전트·기여자 공통의 요구사항·아키텍처 원칙·접근성 상세.
- **`docs/cursor.md`**(본 파일): Cursor 작업 요약, 로컬 명령, **포맷·린트 필수 절차**.

포맷·린트와 같은 워크플로우는 **본 파일**에서 관리한다. 그 외 에디터 보조 규칙이 필요하면 `.cursor/rules`에 둘 수 있으나 **`docs/agent.md`**·본 파일과 **중복·충돌하지 않게** 한다.
