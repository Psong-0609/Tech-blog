# Tech-blog — 에이전트 작업 지침

이 저장소는 **개인/기술 블로그**입니다. 코드 변경 시 아래 기능·비기능 요구사항과 원칙을 따릅니다.

## 기술 스택 (현재)

- Next.js(App Router), React, TypeScript
- 스타일: `@emotion/react` (추가 라이브러리 도입 시 이 문서와 `docs/cursor.md`에 근거 명시)
- **본문(마크다운) 편집**: [TipTap](https://tiptap.dev/) 기반 에디터. 저장·로드 시 **Markdown 문자열**을 기준 형식으로 두고, TipTap 확장·직렬화(또는 MD import/export) 방식은 구현과 함께 **해당 기능이 들어 있는 `src/app` 경로 근처**(또는 주석/README)에 간단히 남긴다.
- **글 소스 저장 위치**: 블로그 본문·메타데이터의 원본은 **`src/` 밖**, 저장소 루트 **`content/`** 아래 **`.md` 파일**로만 둔다(`src/content` 등 코드 트리 안에 두지 않는다). 파일명·하위 디렉터리 규칙·프런트매터 형식은 구현 시 한 가지로 고정하고 코드와 맞춘다.

스택 변경이 필요하면 PR/커밋 설명에 이유와 마이그레이션 범위를 적습니다.

## 기능적 요구사항

### 1. 블로그 글 작성 / 수정 / 삭제

- **작성·수정**: 제목·발행일·카테고리 등 메타데이터와 본문을 **`content/`의 해당 `.md` 파일**에 반영한다. 편집 UI에서는 **TipTap**으로 다루고, 저장 시에는 동일 내용을 **Markdown 파일**로 기록한다(에디터 바깥·빌드·검색 경로는 가능하면 파일/파싱 결과만 사용한다).
- **삭제**: 해당 `.md` 파일 제거(또는 보관 폴더 이동)로 처리한다. 소프트 삭제 여부는 구현 단계에서 결정하되, **실수 방지**(확인 UI, 복구 가능 정책)를 고려한다.

### 2. 카테고리 정리

- 글과 카테고리 관계를 명확히 모델링한다(예: 다대일, 계층 여부는 요구사항 확정 후 반영).
- 카테고리 목록·글 목록 UI에서 일관된 필터/네비게이션을 제공한다.
- 카테고리 편집이 필요하면 글 편집과 같은 편집 흐름·권한 모델에 맞춘다(세부는 구현 시 확정).

### 3. 검색 기능 (카테고리, 검색어)

- **카테고리 필터**와 **키워드 검색**을 조합할 수 있게 한다(URL 쿼리 등으로 상태 공유 가능하면 우선).
- 검색 대상은 원칙적으로 `content/**/*.md`에서 읽거나 빌드 시 생성한 인덱스로 통일하고, 전략은 코드 근처에 짧게 문서화한다.

### 4. 다크모드

- 시스템 선호(`prefers-color-scheme`)와 사용자 선택을 모두 존중하고, 깜빡임(FOUC)을 줄인다.
- 색상 토큰(배경·텍스트·링크·포커스 링)을 중앙에서 관리해 유지보수성을 높인다.

## 비기능적 요구사항

### 1. 성능

- 이미지·폰트·레이아웃 시프트를 줄인다(`next/image` 등 적극 활용).
- 목록·상세에 적절한 캐시·ISR·재검증 전략을 두고, 불필요한 클라이언트 번들을 피한다.
- Lighthouse/Core Web Vitals 악화 변경은 근거와 완화책을 남긴다.

### 2. 유지보수성 (현재 폴더 기준)

- **도메인별 전용 최상위 폴더**(`src/posts` 등)로 나누지 않고, **지금 저장소의 Next.js App Router 레이아웃**을 기준으로 확장한다. 기본 축은 `src/app/`(라우트·`layout`·`page`·필요한 `route` handler)이다.
- 글 원본 디렉터리 **`content/`**는 예외로, **`src/` 밖의 Markdown 소스**로 둔다(코드와 데이터 위치를 혼동하지 않도록 `content/` 경로 참조는 소수의 유틸·레이어에서만 모은다).
- 기능을 추가할 때도 **새 라우트 세그먼트·동일 트리의 colocation**(`page.tsx` 옆 컴포넌트 등)을 우선한다. 여러 라우트에서 쓰는 코드만 공용 위치(예: `src/app/components`, `src/util` 등)로 **필요할 때만** 올린다. 이름 규칙·배치는 **이미 있는 패턴**을 따른다.
- 파일이 비대해지면 같은 기능 단위로만 쪼개고, **순환 참조**는 피한다.
- **함수 순수성(레이어링)**: 의존 관계에서 **하위·리프에 가까운 코드**(파싱·변환·정렬·Markdown 조립 등)일수록 **순수 함수**를 지향한다(동일 입력 → 동일 출력, 참조 투명). 파일·네트워크·환경·전역 변경 같은 **부수 효과는 상위 경계**(예: 라우트·서버 액션·저장 전용 모듈)에 모으고, 하위 레이어는 **인자로 받은 값만** 처리한다. UI 컴포넌트에 묶인 무거운 로직은 가능하면 순수 함수로 끌어낸다.

### 3. 웹 접근성 (ARIA)

- 랜드마크(`main`, `nav`), 제목 계층(`h1`–`h3`), 폼 레이블을 준수한다.
- 상호작용 요소는 키보드로 동작해야 하며, 포커스 상태가 보여야 한다.
- 상태 변화(검색 결과 없음, 로딩, 오류)에는 적절한 **`aria-live`** 또는 명확한 메시지 영역을 둔다.
- 장식용 요소에 불필요한 `aria-*` 남발은 피하고, [공식 패턴](https://www.w3.org/WAI/ARIA/apg/)에 맞게 최소·정확하게 사용한다.

## 에이전트 작업 규칙

- 요구사항과 무관한 대규모 리팩터는 피하고, 변경 범위를 최소화한다.
- 접근성·성능에 영향을 주는 수정은 커밋/PR 설명에 **영향 요약**을 적는다.
- 사용자에게 새 마크다운 문서를 임의로 늘리지 않는다. 단, **`docs/agent.md`**·**`docs/cursor.md`**·**`content/`의 게시용 `.md`**는 각각 프로젝트·블로그 본문 관리 목적으로 유지한다.
